## コンセプト

消防団のボランティア(ボランティア消防士)として **出動するとき、以前トレーニングした手順を思い出せない** 瞬間を解く。

訓練の頻度は低く、出動は不定期で、現場は時間圧と緊張に満ちている。
だから「知っているはずのこと」が、いちばん必要な瞬間に出てこない。

現場タイプを 1 つ選ぶと、自分が訓練した手順を要点だけ 1 画面で出す。
終わったら 1 行記録して積み上げる。それだけのシングルページ web アプリ — **出動リコール** が VPC v1 の Product。

> week02 で「打つ/ペーストの速さ」を扱った。今回は出動時の **"思い出す速さ"** を扱う。

> ⚠️ これは **個人の訓練リコール補助** であり、正式な指揮系統・標準手順・公式マニュアルを置き換えるものではない。

- ★ Gain Creator: 訓練するたび 1 行記録 → 履歴が累積・可視化

## なぜ web3 概論の課題に?

第3回の意図は、**GitHub という場所に慣れる** こと。
これからは AI / AI エージェントと一緒に物を作る時代で、その作業はほぼ全部 GitHub に集まる。

そして、Web3 の本質は **「自分の活動の記録を、自分のものとして積み上げ続ける」** こと。
今回の Product「出動リコール」は、まさに
**自分の訓練・出動の記録を自分の手元(まずは LocalStorage)に積み上げ、必要な瞬間に引き出す** ことが中心テーマ。

さらにこの題材は Web3 と相性が良い:
**訓練・出動の履歴を改ざんできない形でオンチェーンに残せば、それ自体が「訓練の証明」になる。**
「積み上げを自分のものにする」という思想が、訓練実績の信頼性と直結する。
オンチェーン化の判断は v2 以降に持ち越すが、コアの思想は最初の Product に既に入っている。

## 本リポでの運用ルール(公式手順との差分)

公式ルールでは「バグリストは AI に聞かない・自分の頭で・自分の言葉で」とされているが、
本リポでは **AI を共同発散ツールとして使う方針** に変更している。

公式の意図(=自分の文脈は自分しか知らない)は、
**最終確定を本人が行う** ことで担保している(出てきた候補を取捨選択・加筆)。

また、Antigravity を使わず、Claude Code で
- バグリストの発散補助
- VPC キャンバスの SVG 生成
- ファイル構成の整形

を行った。push は手動。
